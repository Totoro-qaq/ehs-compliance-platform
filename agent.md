# EHS Agent 第三层建设方案

## 1. 建设目标

第三层 Agent 不直接替代现有评价、检测、公司管理模块，而是在这些模块之上做一个受控的“合规工作副驾驶”。第一阶段先做可上线、可审计、可扩展的只读 Agent，后续再逐步开放需要确认的写操作。

核心目标：

- 让用户用自然语言查看当前工作台、待处理事项、评价任务、检测报告、限值库和整改建议。
- Agent 的所有数据访问都通过后端工具层完成，不允许模型直接访问数据库。
- 所有工具调用必须记录日志，后续可追溯“谁在什么时候问了什么、调用了哪些工具、返回了什么摘要”。
- 普通用户只能访问自己所属公司的数据，管理员可查看全量或按公司筛选。
- 模型不可用时系统仍可返回规则化摘要，避免前端出现“连不上后端”式体验。
- 先兼容本地 Ollama + Qwen2.5，后续可以切换 Dify、OpenAI、千问 API 或企业私有模型。

## 2. 当前系统定位

现有系统已经具备两层能力：

1. 业务数据层：
   - 公司管理。
   - EHS 评价任务。
   - 检测合规报告。
   - 限值库。
   - 账号、角色、公司隔离。

2. AI/自动化处理层：
   - 评价任务可通过工作流分析材料。
   - 检测报告可解析并运行限值判定。
   - Dify 已经承担过外部工作流 API 的角色。

第三层 Agent 要做的是“把这些能力串起来”，让用户少点页面、少查列表、少手工归纳。

## 3. 第一阶段边界

第一阶段只做只读 Agent，不做自动写库。

允许：

- 汇总工作台数据。
- 查询评价任务列表和详情摘要。
- 查询检测报告列表和结果摘要。
- 查询限值库。
- 根据已有结果生成整改建议。
- 给用户指出下一步应该去哪个页面处理。
- 记录会话、消息、运行记录和工具调用。

禁止：

- 自动创建评价任务。
- 自动删除任务或报告。
- 自动修改公司信息。
- 自动写入限值库。
- 自动重跑任务。
- 绕过后端权限直接读写数据库。
- 让模型自己拼 SQL。

后续写操作必须走二次确认，例如：

1. Agent 生成“建议操作”。
2. 前端展示确认卡片。
3. 用户点击确认。
4. 后端再次校验权限。
5. 后端执行已有 Service 方法。
6. 记录确认和执行结果。

## 5. 模型选型

### 5.1 本机当前建议

你的机器是 i5-13500H、约 34GB 内存、Intel Iris Xe，没有独立 NVIDIA 显卡。这个配置适合跑轻量本地模型和向量检索，不适合长上下文大模型重推理。

推荐本地模型：

- 对话模型：`qwen2.5:7b`
- 备用轻量模型：`qwen2.5:3b`
- 嵌入模型：`bge-m3`

说明：

- `qwen2.5:7b` 可以做中文业务问答、摘要和简单工具结果归纳，CPU 下速度不会特别快，但能用于 MVP。
- `qwen2.5:3b` 更快，适合低延迟演示，但复杂推理和长文本质量会弱一些。
- `bge-m3` 适合中文法规、标准、报告片段的向量化，后面接 Milvus 更合适。

### 5.2 Dify 和 Ollama 的关系

如果 Dify 里已经配置了 BGE，并且你只通过 Dify 做知识库检索，那么不一定要在本地 Ollama 再 pull 一份 BGE。

但如果本系统后端要直接写入和检索 Milvus，就需要后端自己具备嵌入能力，此时建议本地准备 `bge-m3`：

```bash
ollama pull bge-m3
```

阶段建议：

- MVP：先用 Ollama 的 `qwen2.5:7b` 做对话归纳，RAG 暂不强依赖。
- 第二阶段：本系统后端接 Milvus + `bge-m3`，法规知识库从 Dify 外部 API 逐步迁回来。
- 第三阶段：Dify 保留为可选工作流编排，不作为核心法规检索唯一来源。

## 6. 数据库设计

### 6.1 agent_sessions

用途：保存一次 Agent 会话。

字段：

- `id`：UUID。
- `account_id`：发起账号。
- `organization_id`：会话所属公司，普通用户使用账号绑定公司，管理员可为空。
- `title`：会话标题，默认取首条问题前 40 个字符。
- `status`：`OPEN` / `ARCHIVED`。
- `last_message_at`：最后消息时间。
- `created_at` / `updated_at` / `deleted_at`：审计字段。

### 6.2 agent_messages

用途：保存用户和助手消息。

字段：

- `id`：UUID。
- `session_id`：所属会话。
- `role`：`USER` / `ASSISTANT` / `SYSTEM` / `TOOL`。
- `content`：消息正文。
- `metadata_json`：扩展信息，例如引用工具、降级原因、前端展示类型。
- `created_at` / `updated_at` / `deleted_at`。

### 6.3 agent_runs

用途：保存一次模型运行。

字段：

- `id`：UUID。
- `session_id`：所属会话。
- `account_id`：调用账号。
- `organization_id`：调用范围。
- `user_message_id`：触发消息。
- `assistant_message_id`：回答消息。
- `provider`：`ollama` / `dify` / `openai` / `fallback`。
- `model_name`：例如 `qwen2.5:7b`。
- `status`：`RUNNING` / `SUCCEEDED` / `FAILED`。
- `error_message`：失败原因。
- `started_at` / `finished_at`：运行时间。
- `created_at` / `updated_at` / `deleted_at`。

### 6.4 agent_tool_calls

用途：保存每个工具调用的审计日志。

字段：

- `id`：UUID。
- `run_id`：所属运行。
- `session_id`：所属会话。
- `tool_name`：工具名称。
- `arguments_json`：工具参数。
- `result_json`：工具返回摘要。
- `success`：是否成功。
- `error_message`：失败原因。
- `elapsed_ms`：耗时。
- `created_at` / `updated_at` / `deleted_at`。

## 7. API 设计

### 7.1 创建会话

```http
POST /api/v1/agent/sessions
```

请求：

```json
{
  "title": "今天工作台有什么要处理"
}
```

返回：

```json
{
  "id": "...",
  "title": "今天工作台有什么要处理",
  "status": "OPEN",
  "organization_id": "...",
  "last_message_at": "..."
}
```

### 7.2 会话列表

```http
GET /api/v1/agent/sessions?page=1&page_size=20
```

普通用户只返回自己的会话，管理员返回自己的管理员会话。

### 7.3 消息列表

```http
GET /api/v1/agent/sessions/{session_id}/messages
```

返回会话内消息，用于前端恢复聊天记录。

### 7.4 发送消息

```http
POST /api/v1/agent/sessions/{session_id}/messages
```

请求：

```json
{
  "content": "帮我看看今天有哪些待处理事项"
}
```

返回：

```json
{
  "session": {},
  "user_message": {},
  "assistant_message": {},
  "run": {},
  "tool_calls": []
}
```

### 7.5 快捷聊天

```http
POST /api/v1/agent/chat
```

请求：

```json
{
  "session_id": null,
  "content": "帮我总结一下当前工作台"
}
```

如果 `session_id` 为空，后端自动创建新会话。这个接口方便工作台嵌入。

## 9. 权限设计

权限原则：

- Agent 工具必须复用现有 `CurrentUser`。
- 普通用户必须有 `organization_id`。
- 普通用户只能读取本公司评价任务和检测报告。
- 管理员可以跨公司读取，但第一阶段前端不做跨公司筛选。
- 工具层禁止接收前端伪造的 `organization_id` 覆盖普通用户范围。
- 会话、消息、运行记录都绑定 `account_id`。
- 读取历史会话时必须校验会话所属账号。

## 10. Prompt 设计

系统提示词原则：

- 你是 EHS 合规平台内的工作助手。
- 只根据工具返回的数据回答。
- 不要编造不存在的任务、标准、法规条款和检测结果。
- 对不确定的内容明确说“需要人工复核”。
- 对写操作只提出建议，不声称已经执行。
- 输出中文，结构清晰，优先给下一步动作。

简化提示词：

```text
你是 EHS 合规管理平台助手，服务对象包括企业和第三方检测机构。
你只能基于后端工具提供的数据回答，不允许编造任务、报告、标准或法规条款。
当前阶段你只能做只读分析，不能声称已经创建、删除、修改或重跑任何业务数据。
如果数据不足，请说明需要用户进入对应页面复核。
回答要简洁、具体，并给出可执行的下一步建议。
```

## 11. 前端设计

第一阶段放在工作台，不单独做复杂 Agent 页面。

工作台新增“AI 合规助手”卡片：

- 展示当前模型状态：本地模型 / 规则摘要 / 不可用。
- 提供输入框。
- 提供快捷问题：
  - “总结当前工作台”
  - “有哪些待处理事项”
  - “最近失败的任务是什么”
  - “检测报告还有哪些没判定”
- 展示最近一轮回答。
- 回答中给出跳转建议，例如“进入评价任务”“进入检测合规”。

后续再扩展：

- 独立 `/agent` 页面。
- 会话列表。
- 多轮消息记录。
- 工具调用过程展示。
- 引用来源卡片。
- 写操作确认卡片。

## 12. Milvus 知识库规划

### 12.1 为什么要做本地知识库

Dify 适合快速搭 API 和工作流，但长期商业化需要掌握数据治理能力：

- 法规原文版本可控。
- 切片策略可控。
- 向量模型可控。
- 检索日志可审计。
- 可以把限值库、历史报告、整改模板统一纳入检索。

### 12.2 Milvus Collection 建议

第一批 Collection：

- `ehs_regulations`：法规、标准、条款。
- `ehs_limit_references`：限值库条款和依据。
- `ehs_report_snippets`：历史报告可复用片段。
- `ehs_rectification_templates`：整改建议模板。

核心字段：

- `id`。
- `organization_id`：公共法规可为空，企业私有资料必须绑定公司。
- `doc_type`：regulation / standard / report / template。
- `source_name`：来源文件或标准名称。
- `standard_code`。
- `clause`。
- `content`。
- `content_hash`。
- `embedding`。
- `metadata_json`。
- `created_at`。

### 12.3 0 字符文档问题

Dify 知识库里显示可用但 0 字符，建议不要直接依赖。后续本地重建一次更稳。

处理步骤：

1. 收集原始法规 PDF/Word/HTML。
2. 用本系统后端或独立脚本抽取文本。
3. 过滤 0 字符、乱码、纯目录和页眉页脚。
4. 按标准号、章节、条款切片。
5. 用 `bge-m3` 生成向量。
6. 写入 Milvus。
7. 建立抽样验收集，测试常见问题是否能检索到正确条款。

## 13. 商业化视角

Agent 不能只做“聊天窗口”，商业价值应该体现在：

- 降低 EHS 人员查资料和比对任务的时间。
- 让管理层快速知道当前风险和待办。
- 让审计链路可追溯，减少“AI 黑箱”顾虑。
- 把法规、限值、报告、整改建议沉淀为企业知识资产。
- 支持私有化部署，满足企业数据合规要求。

可包装的商业功能：

- 合规工作台智能摘要。
- 风险优先级排序。
- 检测超标解释和整改建议。
- 法规条款检索问答。
- 整改闭环自动草稿。
- 月度/季度合规简报。
- 多公司集团看板。
- 专家复核留痕。

## 16. 风险和控制

风险：模型幻觉。

控制：只把工具数据传给模型，提示词禁止编造；回答中说明数据来源和不确定项。

风险：越权访问。

控制：工具层统一使用 `CurrentUser`，普通用户强制按 `organization_id` 过滤。

风险：本地模型慢。

控制：限制上下文和工具结果数量，失败后规则降级。

风险：数据库表未迁移。

控制：上线前执行 Alembic migration，前端对接口失败给出明确提示。

风险：知识库 0 字符或脏数据。

控制：第二阶段本地重新抽取、切片、验收，不直接把 Dify 历史知识库当可信来源。

## 17. 当前施工决策

本次先施工阶段 A：

- 建表：sessions / messages / runs / tool_calls。
- 接口：sessions、messages、chat。
- 工具：工作台摘要、评价列表、检测列表、限值搜索。
- 模型：优先 Ollama `qwen2.5:7b`，失败自动规则降级。
- 前端：工作台卡片式接入，不先做独立 Agent 页面。

这样可以先把“第三层”的骨架跑通，同时不破坏现有评价、检测和公司管理模块。
