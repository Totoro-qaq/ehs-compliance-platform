# EHS 系统 Agent 与可观测性开发建议

## 目标

本建议面向当前 EHS 合规评价系统的下一阶段演进：在不引入额外法律风险、不显著增加运维复杂度的前提下，逐步增强 Agent 能力、复杂评价报告生成能力、RAG 检索能力和链路可观测性。

核心原则：

- 先可控，再智能。
- 先可验证，再自动化。
- 先做接口壳，再接真实知识库。
- 标准、导则、限值数据不随仓库交付，只连接用户自有或已授权数据源。

## 当前判断

当前系统已有：

- Agent 会话、消息、运行记录、工具调用记录。
- `AgentService` / `AgentTools` 的工具增强问答能力。
- 评价任务、检测报告、标准 chunk、限值查询等业务工具。
- `request_id`、`trace_id`、`span_id`、`traceparent` 基础链路上下文。
- API、Celery、Dify 调用日志串联能力。

当前系统还不是完整自主 Agent loop，更接近：

```text
业务工具增强型助手
```

这对当前阶段是合理的，因为风险低、边界清晰。

## Agent Loop 能力缺口

完整 Agent loop 包含：

```text
用户目标 -> 任务分解 -> 感知环境/上下文 -> 决策 LLM -> 调用工具/API
-> 观察结果 -> 评估/反思 -> 循环
```

当前覆盖情况：

```text
用户目标：有
感知环境/上下文：有，依赖 AgentTools 查询业务数据
决策 LLM：有，但偏单轮
工具调用：有
观察结果：有，tool_results 和 tool_calls 可入库
任务分解：弱
评估/反思：基本没有
循环：基本没有
```

建议不要立即引入复杂 autonomous agent 框架。应先补齐可控性、验证和审计能力。

## Agent Harness

Agent Harness 有必要优先建设。它不是运行框架，而是回归验证框架。

建议新增：

```text
tests/agent_harness/
fixtures/agent_harness/*.json
scripts/run_agent_harness.py
```

Harness 应覆盖：

- 固定业务 prompt。
- 预期调用哪些工具。
- 不允许调用哪些工具。
- 不允许跨公司、跨项目、跨租户读取数据。
- 不允许编造法规、导则、标准条文、报告结论。
- 限值查询、标准依据查询、失败任务总结、待判定报告总结等高频场景。
- 记录 latency、tool_calls、degraded、fallback、error。

优先验证只读场景，再扩展到草稿生成场景。

## Agent Sandbox

当前系统有业务权限隔离，但还不是真正的 Agent Sandbox。

建议先做业务级 sandbox，不要一开始做容器级隔离。

新增运行策略：

```text
AgentRuntimePolicy
```

核心字段：

```text
allowed_tools
max_tool_calls
max_iterations
max_tokens
timeout_seconds
read_only
organization_id
account_id
can_write
can_export
requires_human_approval
```

工具分级：

```text
Level 0：只读查询，例如任务、报告、标准 chunk、限值
Level 1：生成草稿，例如报告章节、整改建议
Level 2：写业务数据，例如创建任务、重跑任务、更新报告
Level 3：外部副作用，例如发邮件、导出正式报告、提交审批
```

当前阶段默认只允许 Level 0 和 Level 1。Level 2 / Level 3 必须人工确认。

重点防护：

- 跨公司读取数据。
- 引用未授权导则/标准。
- 把草稿说成正式结论。
- 编造条文、限值、检测结果。
- 把某客户历史报告内容泄露给其他客户。
- Prompt 注入诱导调用高风险工具。

### Agent Sandbox 施工细化

当前阶段先做运行时策略，不做容器级隔离。

落地边界：

```text
AgentRuntimePolicy.from_actor(actor)
默认 read_only = true
默认 can_write = false
默认 can_export = false
默认 max_iterations = 1
默认 max_tool_calls = 12
默认 timeout_seconds = 30
allowed_tools 从 ToolRegistry 中筛选 READ + NONE + agent_enabled + 不需要审批的工具
```

每次 Agent 运行检查：

```text
运行是否需要人工确认
当前轮次是否超过 max_iterations
每次工具调用前检查 timeout_seconds
每次工具调用前检查 max_tool_calls
每次工具调用前检查 allowed_tools
只读策略禁止 WRITE / EXPORT / ADMIN / EXTERNAL 副作用工具
```

当前不做：

```text
容器隔离
多轮自主循环
写操作审批流
正式导出审批流
工具级 CPU / 内存隔离
```

这一步的目标是先把“Agent 一次运行可以做什么、不可以做什么”收敛到可测试代码里。

## 工具调用体系

当前工具调用足够 MVP，但复杂报告阶段需要升级为：

```text
ToolSpec
ToolRegistry
ToolExecutor
ToolPolicy
ToolCallLog
```

每个工具必须声明：

```text
name
description
input_schema
output_schema
permission_level
side_effect_level
timeout_seconds
rate_limit
requires_approval
allowed_roles
tenant_scope
```

建议工具类型：

```text
READ：只读查询
DRAFT：生成草稿
WRITE：写业务数据
EXPORT：导出或发送正式产物
ADMIN：系统管理
```

示例：

```text
search_standard_chunks        READ
list_assessment_tasks         READ
get_detection_report          READ
generate_report_section       DRAFT
create_assessment_task        WRITE，需要确认
export_official_report        EXPORT，需要确认
import_standard_manifest      ADMIN，禁止 Agent 直接调用
```

工具调用三道防线：

- Pydantic schema 校验入参。
- 角色、组织、项目范围权限校验。
- 记录输入摘要、输出摘要、耗时、错误、trace_id。

### Tool Schema Enforcement 施工细化

当前 `ToolRegistry + ToolPolicy` 已经定义工具边界，下一步要把 schema 从“声明”升级为“强制执行”。

建议先做入参校验，输出只做基础结构校验：

```text
每个工具声明 Pydantic 入参模型
run_tool() 先校验工具名和入参，再执行权限策略，再进入工具实现
非法字段、非法枚举、超出 limit 范围统一返回 AGENT_TOOL_ARGUMENT_INVALID
工具输出必须是 dict；复杂输出 schema 后续在报告流水线阶段再细化
```

优先覆盖：

```text
get_workbench_summary              NoArgs
get_client_project_context         query + limit
list_assessment_tasks              status + context_query + limit
list_detection_reports             status + report_type + context_query + limit
summarize_detection_compliance     context_query + limit
search_regulatory_limits           query + limit
search_standard_chunks             query + filters + include_sensitive + limit
```

测试必须覆盖：

```text
未知工具
未登记工具
额外参数
非法 limit
非法枚举
高风险工具被 Agent 禁止调用
合法参数被规范化后进入工具实现
```

### LLMProvider 最小抽象

Provider 抽象先只负责模型传输，不负责 Agent 编排。

边界：

```text
AgentService 负责组装 system prompt、history、tool_payload
AgentModelProvider 只负责 generate(messages) -> text
OllamaAgentModelProvider 负责 Ollama HTTP 调用
MockAgentModelProvider 只用于测试和本地 harness
未知 provider 返回 AGENT_PROVIDER_NOT_SUPPORTED，由 AgentService 走现有 fallback
```

暂不做：

```text
多模型路由
成本统计
流式输出
函数调用协议
provider 自动重试
跨厂商参数归一化
```

### 暂缓 LangGraph 化和完整多 Provider 产品化

当前 Agent 还是单轮工具增强问答。现在引入 LangGraph 会增加状态图、checkpoint、节点编排和运行时依赖，不符合“先可控、再智能”的阶段目标。

只有出现以下需求时再考虑 LangGraph：

```text
多轮 plan / execute / observe / retry
人工审批节点
可恢复长任务
章节级报告生成流水线
跨工具循环和错误恢复
```

多 provider 切换也只做最小接口，不做产品化配置页。当前阶段只保留 `ollama` 和 `mock` 两类实现，后续接 OpenAI、Claude、RAGFlow 相关能力时再扩展。

## 记忆管理

不建议一开始做“长期人格记忆”。本系统更需要业务记忆。

建议记忆分层：

```text
L0 当前轮上下文
L1 agent_messages / agent_runs，当前已有
L2 项目级 summary，例如 client_project_memory
L3 可审计知识库，例如标准 chunk、报告 chunk、历史结论
```

记忆类型：

```text
会话记忆：当前对话上下文
任务记忆：某个评价任务或检测报告的处理过程
项目记忆：客户、项目、服务类型、历史问题
标准记忆：已引用过的导则、条款、版本
偏好记忆：用户常用报告格式、措辞、输出粒度
```

建议新增表：

```text
agent_memories
agent_memory_events
```

核心字段：

```text
id
organization_id
account_id
scope_type      # session / project / organization / standard
scope_id
memory_type     # preference / fact / decision / warning / citation
content
source_type     # message / task / report / standard_chunk / human
source_id
confidence
is_verified
expires_at
created_at
updated_at
```

原则：

- 记忆必须带 `organization_id`、`account_id`、`project_id` 等范围。
- 默认不跨公司共享。
- 重要记忆必须有来源。
- 记忆要可删除、可过期、可人工确认。
- LLM 生成内容不能直接进入长期记忆，必须经过规则过滤或人工确认。

### Project Memory / Citation Memory 施工细化

当前阶段只落地可审计业务记忆，优先记录标准 chunk 引用，不做自动长期人格记忆，也不把记忆自动注入 LLM 上下文。

已落地边界：
```text
agent_memories
agent_memory_events
AgentMemoryService
```

自动记录规则：
```text
触发来源：search_standard_chunks 工具成功调用后
scope_type：SESSION
scope_id：agent_session.id
memory_type：CITATION
source_type：STANDARD_CHUNK
source_id：chunk_id，缺失时退回 standard_chunk.id
event source：TOOL_CALL
```

Citation memory 只保存引用元数据：
```text
standard_code
standard_name
clause
page_start
page_end
document_id
chunk_id
standard_chunk_id
source_type
source_id
```

明确不保存：
```text
完整 text_chunk
标准/导则原文
embedding
RAGFlow dataset 导出
未确认的 LLM 长期记忆
跨租户共享记忆
```

后续扩展顺序：
```text
1. 增加项目级 summary memory，但必须有人审或规则校验
2. 增加 memory 删除、过期、人工确认接口
3. 报告流水线中记录章节引用和校验结果
4. 需要真实 RAGFlow 后，再把 provider metadata 写入 citation metadata
```
## Trace 与 Jaeger

Jaeger / OpenTelemetry 建议优先于 Prometheus。

当前已有基础：

```text
request_id
trace_id
span_id
traceparent
API 日志
Celery 任务传 trace_id
Dify 出站 traceparent
```

下一步缺的是 span：

```text
agent.run
agent.plan
agent.tool_call.search_standard_chunks
agent.tool_call.list_assessment_tasks
agent.llm.call
rag.retrieve
rag.rerank
report.generate_section
report.verify_citations
report.final_review
```

Jaeger 能回答：

- 复杂评价报告为什么慢。
- 卡在 RAG 检索、LLM 生成、数据库查询还是 Celery。
- 某次 Agent 调用了哪些工具。
- 某段报告用了哪些标准/导则 chunk。
- Dify / Ollama / RAGFlow 调用失败在哪里。

建议接入：

```text
FastAPI instrumentation
httpx instrumentation
SQLAlchemy instrumentation
Celery trace context 传递
OTLP exporter -> Jaeger 或 OpenTelemetry Collector
```

### Trace 施工细化

当前阶段先做本地可测的 tracing 抽象，不直接依赖 Jaeger 服务。

落地边界：

```text
app/core/tracing.py
start_span(name, attributes)
capture_spans() 仅用于测试
span 进入时临时切换当前 span_id
span 退出时恢复父 span_id
span 完成后写日志并可被测试捕获
```

已优先覆盖：

```text
agent.runtime_policy
agent.tool_call.<tool_name>
agent.llm.call
agent.run
```

当前不做：

```text
真实 OTLP exporter
Jaeger Docker / Collector 编排
SQLAlchemy 自动 instrumentation
httpx 自动 instrumentation
采样策略
跨服务 baggage
```

后续接 Jaeger 时，只需要把 `CompletedSpan` 转换为 OpenTelemetry span，业务侧继续使用 `start_span()`。

## Prometheus

Prometheus 当前优先级不高。

它适合回答：

```text
QPS
错误率
P95 / P99
队列积压
资源容量趋势
告警
```

当前阶段已有日志和 trace 上下文，且部署规模较小。完整 Prometheus + Grafana + exporter 会增加运维面。

建议：

- 先不引入完整 Prometheus。
- 后续需要线上告警时，再加轻量 `/metrics`。
- Celery 队列积压、API 错误率、LLM 调用失败率是未来优先指标。

## RAGFlow 接入策略

RAGFlow 有法律风险时，可以只做“壳”，而且建议先只做壳。

正确边界：

```text
RAGFlow = 用户自有/授权标准与导则的解析、切片、检索服务
本系统 = 权限、任务、报告、审批、Agent 工具、业务状态、引用审计
```

不要在仓库中包含：

```text
标准/导则 PDF
法规原文
切片 JSON
embedding dump
RAGFlow dataset export
内置标准库
官方条文示例长文本
真实限值表
```

只做接口抽象：

```text
app/services/rag/provider.py
app/services/rag/ragflow_client.py
app/services/rag/schemas.py
```

Provider 能力：

```text
search(query, filters) -> chunks
get_document(document_id)
get_clause(standard_code, clause)
healthcheck()
```

配置项：

```text
RAGFLOW_BASE_URL
RAGFLOW_API_KEY
RAGFLOW_DATASET_IDS
RAGFLOW_TIMEOUT_SECONDS
```

Agent 工具只暴露：

```text
search_guideline_chunks
get_guideline_clause
```

返回结构必须包含：

```text
standard_code
standard_name
clause
page
version
effective_date
source_uri
chunk_text
score
dataset_id
document_id
```

数据库建议只存引用：

```text
rag_provider
dataset_id
document_id
chunk_id
standard_code
clause
page
score
retrieved_at
```

不要默认长期持久化完整 `chunk_text`。如需缓存，应设置 TTL。正式报告中保存“引用元数据 + 用户确认后的摘录”。

UI 和文档需提示：

```text
请仅连接已获得合法授权的标准/导则数据源。
系统不会校验资料授权状态。
AI 生成内容仅为草稿，正式评价结论需专业人员复核。
```

### RAGFlow 壳施工细化

当前阶段只接只读检索壳，不接导入、解析、删除、重建索引等写操作。

已落地边界：
```text
app/services/rag/schemas.py
app/services/rag/provider.py
app/services/rag/ragflow_client.py
GET /api/v1/ragflow/health
GET /api/v1/ragflow/chunks/search
GET /api/v1/ragflow/clauses/search
```

配置项：
```text
RAGFLOW_BASE_URL
RAGFLOW_API_KEY
RAGFLOW_DATASET_IDS
RAGFLOW_TIMEOUT_SECONDS
```

Agent 只读工具：
```text
search_guideline_chunks
get_guideline_clause
```

未配置 RAGFlow 时，provider 返回 `configured=false` 和空结果，不主动发起网络请求，也不影响本地标准条文库检索。

当前不做：
```text
上传标准/导则文件到 RAGFlow
创建、删除、更新 dataset
把 RAGFlow chunk_text 长期写入本地库
自动把 RAGFlow 命中结果写入 citation memory
多 provider 产品化配置页面
```

## 复杂评价报告生成建议

几万字评价导则不能一次性丢给 LLM 生成完整报告。建议拆成流水线：

```text
1. 识别项目类型和评价范围
2. 解析资料清单
3. 按章节检索导则
4. 生成章节草稿
5. 校验引用来源
6. 校验结论是否有依据
7. 生成问题清单
8. 人工复核
9. 导出正式报告
```

每个章节都应保存：

```text
输入资料
检索 query
召回 chunk
引用条款
生成草稿
校验结果
人工确认状态
```

正式报告必须区分：

```text
AI 草稿
已人工复核
正式出具
```

## 推荐实施顺序

建议按以下顺序推进：

```text
1. Agent Harness：先把行为测住
2. ToolRegistry + ToolPolicy：把工具边界管住
3. Tool Schema Enforcement + LLMProvider 最小抽象：把工具入参和模型传输测住
4. Agent Sandbox：限制权限、轮次、写操作、导出操作
5. Trace / Jaeger：追踪 Agent、RAG、LLM、Celery
6. Project Memory / Citation Memory：做可审计业务记忆
7. RAGFlow 壳：只接用户自有/授权实例
8. Report Pipeline：按章节生成、引用校验、人工复核
9. Prometheus：有线上告警需求后再加
```

不建议优先做：

- 完整 autonomous loop。
- 自动写库或自动导出正式报告。
- 内置标准/导则知识库。
- 长期人格记忆。
- 大而全的 Prometheus/Grafana 监控栈。

## 总结

下一阶段的核心不是让 Agent 更自由，而是让 Agent 的每一步：

```text
可控
可测
可追踪
可引用
可审计
可人工接管
```

RAGFlow 可以接，但应作为可插拔检索后端，不作为随仓库交付的知识库。Jaeger 比 Prometheus 更适合当前问题定位。Agent Harness、Sandbox、ToolPolicy 和 Memory 边界应先于复杂循环能力建设。
